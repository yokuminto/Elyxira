# 回顾 — P2P 配对同步（路线 B）

**日期**: 2026-06-30

## 一、任务概述

在已有 GitHub 仓库同步（题库 + break 笔记/标签）基础上，新增"分布式/局域网感"的设备间 P2P 同步：手机扫码 6 位 PIN 即与电脑配对，之后 WebRTC DataChannel 直传 Yjs CRDT 更新，断网仍可离线缓存（y-indexeddb），全程 AES-GCM 加密（密钥从 PIN 派生）。

## 三、关键设计决策

### 1. PIN+信令中继，不是 QR 直接传 SDP
浏览器间无法做原始 UDP 广播；纯 QR 传 SDP 需要 scan 多个 QR（ICE candidate 太多）、且仅限 LAN。选 PIN（6位数字）+ Cloudflare Worker Durable Object 充当临时 pub/sub，配对完成后服务器对数据完全无知（y-webrtc password 参数加密 SDP + 载荷）。

### 2. y-webrtc 内置 password = PIN 内嵌加密
y-webrtc 用 PBKDF2(100k iter, roomName salt) 派生 256 位密钥，AES-GCM 加密所有信令 + DataChannel 流量。无需自行实现 crypto 代码，PIN 既是房间号又是密钥源。

### 3. 脱敏 + 远端合并保留本地敏感
`AppSettings.githubConfig.token` 和 `apiPresets[].apiKey` 镜像入 Yjs 前置空串。远端 apply 时 `_mergeRemoteSettingsIntoLocal` 按 name 对齐 apiPresets 复用本地 apiKey、合并 githubConfig 但保留 token。每设备各持自有 PAT，配对中不会泄密。

### 4. 防回环：`_isApplyingRemote`
远端 update 触发的订阅 callback 同步段内设 `_isApplyingRemote=true`，期间 `mirrorToYjs` 自动 skip。否则 callback 写本地 → saveSettings 触发 mirrorToYjs → 写 Yjs 文档 → 远端 apply 死循环。

### 5. y-indexeddb 自建独立数据库
不同 PIN 配对不同 IndexedDB（yjs-<roomName>）。下次用同 PIN 配对可合并 CRDT 历史；新 PIN 视作全新会话，旧的离线状态保留不删（无害遗留）。

## 四、反思与收获

### 做得好
- **分批委派 + V3 验证**：modal 委派 visual-engineering，但我亲自读 modal 完整内容发现 subagent 自报完成但漏写了 `const props = defineProps`（只写 `defineProps<>()` 没赋给 `props` 变量），运行时 ReferenceError。统一在验证阶段抓到并修复。
- **依赖аров.order**：依赖 → 信令 → CRDT 服务 → modal → 入口 → 业务集成，每层 commit 独立可 revert
- **脱敏设计前置**：在写 `_sanitizeSettingsForP2P` 前先盘清楚 AppSettings 的敏感字段位置，避免后期把 token 写上 CRDT 然后回滚
- **零 polyfill 依赖**：y-webrtc 自动被 Vite chunk 切出 112KB lazy，主包不增大；不需要在 main.ts 引入 Buffer/process polyfill

### 可改进
- **modal 委派 prompt 仍不够严**：subagent 在 promise resolve 后没做 bug 自检。下次 prompt 要加 "做一遍 mental run-through，从 props.show 变化开始模拟整个生命周期"
- **snapshotProvider 一开始只支持单 provider**：写 code 时我假设单一注册方，配对集成 breakStorage 与 configService 时发现两边都要 snapshot，被迫 refactor 成 `_snapshotProviders: Set<SnapshotProvider>`。下次设计公开接口一开始就考虑多注册
- **CSS 类的可用性靠经验**：modal-p2p-sync 用 `.modal__settings-group`、`.modal__form-group` 等，CSS 全局类没文档化，靠 grep 同类 modal 才知道命名。`.agent/memory/code_conventions.md` 应该补一条"全局 modal CSS 类清单"

### 待跟进（留作低优先级 backlog）
1. **y-indexeddb 离线数据库堆积**：每次配对一个新 PIN 就多一个 IndexedDB 库；未来加自动清理机制（>30 天未访问自动删）
2. **2-peer 上限**：信令服务器 `if (this.peers.size >= 2) return 409`，多于两台设备不支持；可扩到 mesh（y-webrtc 原生支持 maxConns）
3. **quizData/quizCache_* 不入 CRDT**：它们 500KB-2MB×N，GitHub repo 同步已覆盖；要 P2P 要拼大小优化（或分片 chunk CRDT）
4. **信令 URL 占位**：`DEFAULT_SIGNALING_URL = 'wss://elyxira-signal.workers.dev'` 未部署，用户需自己 `wrangler deploy` 后填入设置
5. **y-webrtc 默认 signaling servers 全死**：开源 y-webrtc 自带 3 个 yjs.dev/heroku 信令全挂，必须自部署；这点要在 README 里再强调
6. **跨网格两次扫 QR 模式**：当前是 PIN+服务器；如果未来想完全 serverless，可参考 QWBP（55-100 字节 binary SDP）走 QR 双向扫描，但仅限 LAN

## 五、技术栈踩坑点

| lib | version | 备注 |
|-----|---------|------|
| yjs | ^13.x | 核心 CRDT 文档 |
| y-webrtc | ^10.x | **自带信令服务器全死，必自部署**；password 参数自动 AES-GCM |
| y-indexeddb | ^9.x | 独立 IndexedDB = `yjs-<roomName>`，不复用业务数据库 |
| qrcode | ^1.x | `QRCode.toCanvas(canvas, text, {width, margin})` 直接画 canvas |

## 六、部署流程（用户必做）

```bash
cd cloudflare-worker
npm i -g wrangler
wrangler login
wrangler deploy
# 得到 wss://elyxira-signal.<子域>.workers.dev
```

UI 内：P2P 配对同步 modal → 高级设置 → 信令服务器 URL → 填入。
