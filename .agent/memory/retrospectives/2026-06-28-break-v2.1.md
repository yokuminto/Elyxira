## 三、反思与收获

### 做得好
- **设计先行**：先写了完整的设计书再实现，避免了边走边改。盒子规则、概率表、兜底链都在设计阶段确定好。
- **5 路并行审查**：goal/QA/code quality/security/context mining 并行跑，发现了手动 review 漏掉的 bug（如 nextQuestion 缺守卫、权重溢出丢失、_drawnIds 初始化位置错误）。
- **最小修复**：重复题的根因分析精准（缺失消费），一行 splice 修复，配合 _drawnIds 去重覆盖跨盒路径。
- **链借机制**：反向利用运行时的兜底链，把缺题的盒从链上借题填充，同时 AI 顺手生成借来题的笔记——一举两得。

### 可改进
- **_drawnIds 初始化时机**：第一次放在首题抽取之后，导致首题 ID 被 wipe。应该在抽题之前初始化——审查第二轮才抓到。
- **预抽池范围问题**：初版 _buildBoxes 扫全题库（10,069 题），但补给站只为预抽池（114 题）生成笔记——大部分抽到的题没笔记。改为扫 preDrawnQuestionIds 后才修复。
- **_sessionPool 直接删除过急**：删了旧的 _drawFromPool 但没有在一开始融入消费语义，中间经历了一次"纯随机有放回"的阶段才补上 splice。
- **设计文档在 .gitignore 里**：.agent/ 目录被忽略，更新了 design_decisions.md 但 push 不上去——后续需要确认是否应该把设计文档纳入版本控制。

### 待跟进

- 复习间隔机制（SM-2/FSRS/Leitner）尚未引入——当前只有盒子分类，没有"什么时候该复习"的调度

## 四、经验总结

### 模块设计
- **存储抽象层**：breakStorage 对外 API 不变（getNotes/setNote/recordAnswer），内部从 localStorage 切 IndexedDB + 自动升迁，调用方零改动
- **三级回退的普适性**：预抽→全量→兜底链→全题库随机 的四层结构可以复用到任何"有优化缓存、需降级兜底"的场景

### 常见陷阱
- **Vue watcher + 对象引用**：`_syncNotesToQuestion` 必须返回 `{ ...q, notes }` 新对象，原地改 `q.notes` 不触发 watcher
- **IndexedDB version upgrade**：新增 store 必须升版本号或做自动升迁（_ensureDB），否则 `onupgradeneeded` 不触发 → 新 store 不存在 → 事务失败
- **有放回 vs 无放回抽取**：新系统的随机取题默认无消费，和旧 _sessionPool shift 消费语义不一致→重复题 bug。切换抽题算法时必须检查消费语义
- **Set 初始化位置**：去重 Set 必须在所有消费之前初始化，否则初期数据丢失

### 可复用模式
- **_ensureDB 自动升迁**：opening → check stores → close → increment version → reopen（任何需要动态升级 IndexedDB schema 的场景通用）
- **安全阀 + 溢出重分配**：先清零空盒权重 → 再按原始比例分配溢出 → 保证溢出不流入空盒消失
- **链借补全**：运行时兜底链反向用于预抽池补全——借来的题进 preDrawnQuestionIds 供 AI 生成笔记，运行时原始兜底逻辑无需改动
- **5 路并行审查**：每轮大改动后 fire 5 个并行审查 agent（goal/QA/code/security/context），总审查时间 = 最慢 agent 的耗时
